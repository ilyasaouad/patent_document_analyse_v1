# Patent AI Detection Analyzer

This skill defines the exact 4-phase heuristic methodology used by the `ai_document_detect.py` pipeline to evaluate whether a patent application (specifically its description, claims, and drawings) was generated by a Large Language Model (LLM) or a human patent attorney.

## Core Objective
To securely and locally process extracted patent Markdown files (`.md`) and output a definitive **Confidence Score** and **Risk Level** by identifying stylistic LLM fingerprints, logical hallucinations, and multimodal inconsistencies.

---

## The 4-Phase Detection Heuristic

The engine runs a sequential algorithm utilizing a local Ollama instance (default: `gpt-oss:120b-cloud`), weighted across four distinct analytical phases.

### Phase 1: Stylometric Fingerprinting (Weight: 30%)
**Target File:** `description*.md`

LLMs exhibit highly specific mathematical writing habits compared to humans. This phase specifically scans the description text for:
1. **Low Burstiness:** Uniform, perfectly structured sentence lengths throughout the document.
2. **Formulaic Transitions:** Repetitive use of terms like *"Furthermore"*, *"Moreover"*, and *"Additionally"*.
3. **Lack of Idiosyncrasies:** The absence of the natural "noise" and personal grammatical quirks typically found in human-drafted legal documents.
4. **Passive Voice Abuse:** Excessive and generalized passive voice structures.

### Phase 2: Anchor Method Comparison (Weight: 40%) - *Strongest Signal*
**Target Files:** `description*.md` & `claim*.md`

This is a self-reflexive test and the single strongest indicator of AI generation.
1. The analyzer extracts the **Claims**.
2. It prompts the LLM to *generate its own synthetic description* based purely on those claims.
3. It then compares its generated "Anchor" description directly against the user's uploaded description.
**Logic:** If the user's uploaded description reads identically (in logical flow and phrasing) to the AI's synthetic anchor, it is highly probable the user simply fed the claims into an LLM and asked it to write the description.

### Phase 3: Technical Hallucination Detection (Weight: 20%)
**Target Files:** `description*.md` & `claim*.md`

Generative AI often attempts to artificially "flesh out" a patent description by inventing arbitrary technical components, dimensions, or specific materials that the author never actually claimed.
This phase strictly cross-references the claims against the description to flag and penalize any unsupported technical novelties introduced solely in the description.

### Phase 4: Drawing Consistency & Alignment (Weight: 10%)
**Target Files:** `description*.md` & `drawing*.md` (OCR)

LLMs struggle heavily with spatial reasoning and numerical consistency across multimodal boundaries.
1. The script extracts OCR data, specific references, and numbers from the uploaded patent drawings.
2. It cross-references these visual labels against the descriptive text.
**Logic:** If the description references "Gearbox 15" but the drawing OCR explicitly labels it "12", or fails to document it entirely, this points to generative hallucinations breaking down when attempting to maintain complex antecedent basis across text and imagery.

*(Note: If the description explicitly references drawings but no drawing file is uploaded, the script immediately flags a critical warning to the user).*

## Execution Flow & Outputs

The script reads inputs dynamically from `document_text_output/`, routing files accurately by prefix (`claim*`, `description*`, `drawing*`). 

### Output Generation (`ai_detection_result.json`)
The final output is compiled into a universally accessible JSON object containing:
- **`confidence_score`**: The calculated 0-100% probability of AI generation.
- **`risk_level`**: A distinct triage tag (`MINIMAL`, `LOW`, `MEDIUM`, `HIGH`) warning the reviewer.
- **`findings`**: Specific strings explaining precisely *why* the text failed the heuristics.
- **`recommendations`**: Actionable steps for the user or examiner (e.g., "Request author revision to reduce formulaic language").

## Configuration Variables
The entire engine dynamically inherits its variables from the global `config/settings.py` file:
- `OLLAMA_MODEL` (e.g., `gpt-oss:120b-cloud`)
- `OLLAMA_URL` (e.g., `http://localhost:11434`)
- `AI_DETECTION_THRESHOLD` (e.g., `0.6`)
- `LLM_API_KEY` (Accepts Bearer tokens for seamless transition to cloud APIs)
